## Project Overview
For this capstone project, I chose to develop a **Simon Says memory game**. The goal of the game is for the player to watch a sequence of colors generated by the computer and repeat the same pattern in the correct order. Each round adds a new color to the sequence, making the game more challenging as the player progresses.

This project was built using **HTML, CSS, and JavaScript** and was designed to demonstrate DOM manipulation, event handling, game logic, conditional statements, arrays, functions, and user interaction.

---

## Development Plan

### Game Chosen
The game I chose to develop was **Simon Says**, a memory-based game where the player must correctly repeat a pattern of colors shown by the computer.

### Planned Functions
The following functions were planned to support the game logic:

- `startGame()` to initialize or restart the game
- `nextRound()` to add a new color to the sequence
- `playSequence()` to show the computer-generated pattern
- `handleUserClick()` to track the player’s input
- `checkUserInput()` to compare the player’s clicks to the computer’s sequence
- `endGame()` to stop the game and show a loss or reset state

### Planned Features
- Start button to begin gameplay
- Four colored game buttons
- Random color sequence generation
- Increasing difficulty each round
- Visual effects when colors flash
- Win/loss or round update messages
- Restart functionality

### Planned Algorithms and Coding Choices
- Use an **array** to store the computer’s pattern
- Use another **array** or tracking method for player input
- Use **Math.random()** to generate the next color
- Use **conditional statements** to determine whether the player clicked the correct color
- Use **event listeners** for player interaction
- Use **setTimeout()** to control timing when displaying the sequence
- Use simple, readable JavaScript logic so the code stays organized and easier to debug

## Implementation Plan

To implement the project, I planned to complete the game in the following order:

1. Create the HTML structure for the title, instructions, start button, and game board.
2. Style the layout using CSS so the game looked visually appealing and easy to use.
3. Add the four Simon game buttons and assign each one a color.
4. Build the JavaScript logic to:
   - start the game
   - generate a random pattern
   - display the pattern to the player
   - accept user clicks
   - compare the player’s input to the correct answer
5. Add text feedback to show whether the player advanced or lost.
6. Test the game repeatedly and fix any bugs in logic, timing, or button interactions.
7. Deploy the finished project to GitHub Pages.

## Reflection on Coding Trade-Offs

While building the project, I made several coding trade-offs.

One trade-off was choosing **simplicity over complexity**. I used basic arrays, functions, and conditionals instead of advanced patterns or frameworks so the code would remain easier to read and maintain.

Another trade-off was balancing **design and performance**. I added creative styling choices such as a beach background, custom font, and glow effects to make the game feel more original and visually interesting. Although these choices improve appearance, they can slightly increase page load time compared to a simpler design.

I also chose to focus on a **clear user experience** rather than adding too many extra features. This helped keep the gameplay straightforward and aligned with the project requirements.

## Justification, Challenges, and Debugging

I chose Simon Says because it is a recognizable game that still requires enough logic to demonstrate core JavaScript skills. It allowed me to practice using arrays, functions, DOM events, timing, and game state.

One challenge was controlling the timing of the computer-generated sequence. At first, the sequence moved too quickly or did not feel organized. I worked through this by using `setTimeout()` so each color could flash in order with a clear delay.

Another challenge was comparing user input to the correct sequence. I had to carefully check that each click matched the correct color at the correct position. Debugging this required reviewing how the arrays were updated and when the round should end.

I also worked through issues involving event handling, making sure the player could only click during the proper part of the game. This helped prevent accidental inputs and made gameplay smoother.

During troubleshooting, I tested the game repeatedly, reviewed the JavaScript logic, and made corrections to improve sequence playback, round progression, and reset behavior.
